Variable Scope

Understand where variables live and where they can be accessed — the rules that govern visibility across global scope, function scope, and block scope.

1

Definition

Scope in JavaScript refers to the accessibility (visibility) of variables in different parts of the program. It determines where a variable can be read or modified.

There are three main types of scope in JavaScript: Global Scope — accessible everywhere; Function Scope — accessible only inside the function where it was declared; and Block Scope — accessible only inside the {} block where it was declared (applies to let and const).

2

Simple Way

Scope answers one question: "Where can I use this variable?"

Global → Declared outside everything — accessible from anywhere in the file
Function → Declared inside a function — only accessible within that function
Block → Declared inside {} — only accessible within that block (let / const)
3

Code Examples

Example 1 — Global Scope 
let name = "Aman"; // global — declared outside everything

function showName() {
  console.log(name); // accessible inside function
}

showName();
console.log(name); // accessible outside too
Output
Click ▶ Run to execute
Explanation
  • name is declared at the top level — it belongs to the global scope
  • Global variables are readable from any function or block in the same file
  • Use global scope sparingly — too many globals make code hard to manage
Example 2 — Function Scope 
function test() {
  let message = "Hello"; // function-scoped
  console.log(message);  // works fine inside
}

test();

// Trying to access outside the function:
try {
  console.log(message); // ReferenceError
} catch (e) {
  console.log("Error:", e.message);
}
Output
Click ▶ Run to execute
Explanation
  • message is created when test() runs and destroyed when it ends
  • Accessing it outside its function throws a ReferenceError
  • Function scope is one of JavaScript's core privacy mechanisms
Example 3 — Block Scope (let & const) 
if (true) {
  let x = 10;
  const y = 20;
  console.log(x, y); // works inside the block
}

// Trying to access outside the block:
try {
  console.log(x);
} catch (e) {
  console.log("Error:", e.message);
}

try {
  console.log(y);
} catch (e) {
  console.log("Error:", e.message);
}
Output
Click ▶ Run to execute
Explanation
  • let and const are block-scoped — they only exist within the {} they were declared in
  • A block is any pair of curly braces: if, for, while, or a standalone {}
  • Accessing them outside throws a ReferenceError
Example 4 — var vs let (Block Scope Difference) 
if (true) {
  var a = 5;  // var IGNORES block scope
  let b = 10; // let respects block scope
}

console.log(a); // 5 — leaks out of the block!

try {
  console.log(b); // ReferenceError
} catch (e) {
  console.log("Error:", e.message);
}
Output
Click ▶ Run — spot the difference!
Explanation
  • var is function-scoped, not block-scoped — it leaks out of if, for, and while blocks
  • This is a common source of hard-to-find bugs — which is why var is avoided in modern JavaScript
  • let and const stay contained to their block — use them instead
4

Real-Life Example

Think of scope like rooms in a house — each has different levels of access:

Global
The living room — open to everyone in the house at any time
Function
A private bedroom — only accessible to whoever is inside it
Block
A locked cupboard inside the bedroom — even more restricted
var
A door left wide open — the variable escapes its room unexpectedly

MCQs

Q1
Which keyword creates a block-scoped variable?
  • A var
  • B let
  • C Both
  • D Neither
💡 B is correct. let (and const) are block-scoped — they stay inside the {} they are declared in. var is function-scoped and leaks out of blocks like if and for.
Q2
What happens when you run this code? 
function test() {
  let x = 10;
}
test();
console.log(x);
Output
Run it to verify
  • A 10
  • B undefined
  • C ReferenceError
  • D null
💡 C is correct. x is declared with let inside test() — it is function-scoped and cannot be accessed outside. JavaScript throws a ReferenceError.
Q3
Where must a variable be declared to have global scope?
  • A Inside a function
  • B Inside an if block
  • C Outside all functions and blocks
  • D Inside a loop
💡 C is correct. A variable declared at the top level of a script — outside any function, block, or loop — belongs to the global scope and is accessible from everywhere in that file.
8

Pro Tips & Extra Knowledge

  • 01

    Keep variables as local as possible — declare them in the smallest scope where they are needed. This avoids accidental overwrites and makes code easier to reason about:

    Prefer local over global 
    // ✗ Unnecessary global
let result;
function calculate() {
  result = 5 + 3;
}
calculate();
console.log(result);

// ✓ Keep it local, return the value
function calculateClean() {
  let result = 5 + 3;
  return result;
}
console.log(calculateClean());
    Output
    Click ▶ Run to execute
  • 02

    Variable shadowing — a local variable can have the same name as a global one. Inside the function, the local version takes priority:

    Shadowing example 
    let x = 10; // global x

function test() {
  let x = 20; // local x — shadows the global
  console.log(x); // 20 — sees the local one
}

test();
console.log(x); // 10 — global is unchanged
    Output
    Click ▶ Run to execute
  • 03

    Never use var in modern JavaScript — its function-only scoping leaks variables out of loops and if blocks in ways that cause subtle bugs:

    var leaking out of a loop 
    // var leaks — i survives after the loop
for (var i = 0; i < 3; i++) {}
console.log(i); // 3 — leaked!

// let is contained — i does not survive
for (let j = 0; j < 3; j++) {}
try {
  console.log(j);
} catch (e) {
  console.log("Error:", e.message); // ReferenceError
}
    Output
    Click ▶ Run to execute
  • 04

    Proper scoping improves memory management — local variables are garbage-collected when their function ends, whereas globals persist for the entire lifetime of the page.

Mini Challenge

There are two variables named x — one global, one local. Which one does each console.log see? Predict both outputs before running.

What does this print? 
let x = 10;

function test() {
  let x = 20;
  console.log(x);
}

test();
console.log(x);
Output
Think first, then run!
Step-by-Step Explanation
// Two variables — same name, different scopes:
//   Global x = 10   (line 1)
//   Local  x = 20   (inside test())

// Step 1: test() is called
//         JavaScript enters the function scope
//         A new local x = 20 is created — it SHADOWS the global x
//         console.log(x) inside sees the LOCAL x
//         Output line 1 → 20

// Step 2: test() ends
//         The local x is destroyed — it no longer exists
//         We are back in the global scope

// Step 3: console.log(x) outside
//         Only the global x = 10 is visible here
//         Output line 2 → 10

// Final output:
//   20   ← local x inside test()
//   10   ← global x, unchanged

// Key takeaway:
//   Shadowing lets inner scopes reuse a name without
//   affecting the outer variable. The two x values
//   are completely independent of each other.
